# 需求解决方法智能体 — 三层架构

本智能体采用三层架构：交互层、AI 推理层、文档生成层。各层职责清晰，便于实现与扩展。

---

## 1. 交互层（Web 对话界面）

**职责**：用户入口、会话管理与下载。

- **文档上传**：支持 Word(.doc/.docx)、Excel(.xls/.xlsx)，在后端或本层完成文本提取后，将纯文本传入 AI 推理层。
- **需求文字输入**：用户在对话框直接输入需求描述，作为消息传入 AI 推理层。
- **会话管理**：创建新会话、按客户区分历史会话、加载历史对话供迭代使用。
- **流式展示**：接收 AI 推理层的流式输出，逐步展示方案内容。
- **下载**：内容生成完成后展示「下载 Word」按钮，调用文档生成层得到文件后提供下载（命名：`解决方案_客户名称_日期.docx`）。

**与本 Agent 的关系**：交互层不承载模型推理，仅负责把用户输入（文本/附件解析结果）发给 solution_agent，并展示其输出与下载入口。

---

## 2. AI 推理层（大模型驱动）

**职责**：需求解析 + 方案生成。即本 Agent（solution_agent）。

- **需求解读**：基于上传文档提取的文本或用户输入的需求描述，理解业务背景、现状、痛点、目标与功能需求。
- **方案生成**：按六章模板（见 document-structure.md）生成完整解决方案正文（Markdown 或结构化文本）。
- **多轮迭代**：根据用户追加的修改意见与对话历史，对方案进行局部或全局修订。
- **流式输出**：若运行时支持，以流式方式返回方案内容，提升等待体验。

**输出**：结构化方案正文，供文档生成层渲染；不直接生成 .docx。

---

## 3. 文档生成层（模板渲染为 Word）

**职责**：将 AI 推理层输出的方案内容渲染为规范 Word 文档。

- **输入**：方案正文（含六章结构）、客户名称、日期等元信息。
- **模板**：固定版式（封面、目录、标题层级、正文样式）。
- **输出**：标准 .docx 文件，命名 `解决方案_客户名称_日期.docx`。
- **触发**：由交互层在「内容生成完成」后调用，生成文件并返回下载链接或二进制流。

**与本 Agent 的关系**：文档生成层为独立服务或模块，不参与对话；仅接收结构化内容与元数据，输出 Word。

---

## 数据流概览

```
用户 → [交互层] 上传/输入 → [AI 推理层] 解读 + 生成/迭代 → 方案正文
                ↑                              ↓
           会话/历史 ←──────────────────── 流式/最终正文
                ↓
         [文档生成层] 渲染 Word → 下载
```

---

## 实现说明

- **交互层**：可为实现为独立 Web 应用（如 React/Vue），通过 Gateway 或后端 API 与 solution_agent 通信。
- **AI 推理层**：使用本仓库的 agent 运行栈（如 agent-from-dir + Gateway），agentId 为 `solution_agent`。
- **文档生成层**：可使用 docx 模板引擎（如 docx、python-docx、docxtpl 等）或低代码文档服务，按 document-structure.md 映射章节到样式。
